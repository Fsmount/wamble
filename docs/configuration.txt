Wamble Configuration
====================

Overview
- Format: minimal Lisp-like expressions with forms and symbols.
- Top-level forms are evaluated in order.
- Use `(def name value)` to define variables.
- Use `(defprofile name ((forms...)))` to define a named profile.

Core Options (common defaults)
- `port` (int, 8888): UDP server port.
- `websocket-enabled` (int, 0): Enable websocket bridge for browser clients
  on this profile/runtime.
- `websocket-port` (int, 0): Websocket TCP port. If set to `0`, it uses `port`
  as the TCP websocket port number.
- `websocket-path` (string, `/ws`): Websocket HTTP upgrade endpoint path.
  - Must start with `/`.
- `experiment-enabled` (int, 0): Enable treatment-group assignment, directed
  opponent-group edges, and treatment outputs for this profile.
- `experiment-seed` (int, 0): Reserved for deterministic experiment behavior
  that depends on a per-profile seed.
- `timeout-ms` (int, 100): Reliable send ACK timeout (ms).
- `max-retries` (int, 3): Reliable send retries.
- `max-message-size` (int, 126): Fixed message body size (informational).
- `buffer-size` (int, 32768): Socket send/recv buffer bytes.
- `max-client-sessions` (int, 1024): Client session capacity.
- `rate-limit-requests-per-sec` (int, 120; `0` disables): Per-token request
  budget applied at runtime.
- `session-timeout` (int, 300): Idle session timeout (sec).
- `max-boards` / `min-boards` (int, 1024 / 4): Board pool sizing.
- `inactivity-timeout` (int, 300): Archive inactive boards (sec).
- `reservation-timeout` (int, 1209600): Board reservation lease (sec).
- `default-rating` (int, 1200): Starting rating for new players.
- `max-players` (int, 1024): Player capacity.
- `token-expiration` (int, 86400): Token TTL (sec).
- `max-pot` (double, 20.0): Scoring pot cap per board.
- `max-moves-per-board` (int, 1000): Move history cap per board.
- `max-contributors` (int, 100): Max contributors per board for scoring.
- `state-dir` (string, unset): Directory for hot-reload state snapshots. If
  unset, the server uses a platform-specific temp location (`/tmp` on POSIX).
- `new-player-early-phase-mult` (double, 2.0): Payout multiplier for new players in the early game.
- `new-player-mid-phase-mult` (double, 1.0): Payout multiplier for new players in the mid game.
- `new-player-end-phase-mult` (double, 0.5): Payout multiplier for new players in the late game.
- `experienced-player-early-phase-mult` (double, 0.5): Payout multiplier for experienced players in the early game.
- `experienced-player-mid-phase-mult` (double, 1.0): Payout multiplier for experienced players in the mid game.
- `experienced-player-end-phase-mult` (double, 2.0): Payout multiplier for experienced players in the late game.
- `db-host`, `db-user`, `db-pass`, `db-name` (string, defaults
  localhost/wamble/wamble/wamble): Database connection.
- `db-port` (int, 5432): Profile game DB port.
- `global-db-host`, `global-db-user`, `global-db-pass`, `global-db-name`
  (string, defaults localhost/wamble/wamble/wamble_global): Process-wide
  global store used for identities, policy rules, and config snapshots/events.
- `global-db-port` (int, 5432): Shared global DB port.
  All profiles in the process must use the same `global-db-*` endpoint, and it
  must be different from every profile's game DB endpoint (`db-host/user/name`).
Database Connections
- Each profile listener runs in its own thread and opens its own DB connection
  using that profile's `db-host`, `db-port`, `db-user`, `db-pass`, `db-name`.
- `select-timeout-usec` (int, 100000): Main loop select timeout (usec).
- `cleanup-interval-sec` (int, 60): Session cleanup cadence.
- `max-token-attempts` / `max-token-local-attempts` (int, 1000 / 100): Token
  generation limits.
- `persistence-max-intents` (int, 128): Upper bound per flush cycle for how
  many queued persistence intents are selected for DB apply.
- `persistence-max-payload-bytes` (int, 65536): Upper bound per flush cycle for
  estimated total payload of selected persistence intents.
- `prediction-mode` (int, 0): Prediction feature mode.
  - `0`: disabled.
  - `1`: only the currently reserved player may predict the next move.
  - `2`: free prediction streak mode; any allowed predictor may predict the
    next half-move.
  - `3`: gated streak mode; mode `2` behaviour plus deterministic session
    gating based on `prediction-gated-percent`.
- `prediction-base-points` (double, 1.0): Base points for a correct prediction.
- `prediction-streak-multiplier` (double, 2.0): Multiplier applied to each
  successive correct streak step in modes `2` and `3`.
- `prediction-streak-cap` (int, 10): Maximum streak depth counted for scoring.
- `prediction-gated-percent` (int, 10): Percent of sessions allowed to predict
  in mode `3`.
- `prediction-view-depth-limit` (int, 2): Default maximum depth exposed when
  reading prediction trees.
- `prediction-penalty-incorrect` (double, 0.0): Optional points deducted for
  wrong predictions.
- `prediction-match-policy` (string, `exact-uci`): Matching mode for predicted
  moves.
  - `exact-uci`: promotion piece must match exactly.
  - `from-to-only`: only the first four UCI characters must match.
- `prediction-max-pending` (int, 64): Maximum in-flight predictions tracked per
  board and prediction-visibility depth bucket; sessions that share readable
  prediction context are evaluated against the same pending budget.
- `prediction-max-per-parent` (int, 1): Maximum pending predictions a single
  session may submit for the same parent branch on a board.
- `prediction-enforce-move-duplicate` (int, 1): When non-zero, reject
  move-identical pending predictions at the same parent branch (subject to
  read-visibility rules). When `0`, move-identical duplicate checks are disabled.
- `chess960-interval` (int, -1): Controls automatic Chess960 board creation.
  - `-1`: disabled (all boards are another mode).
  - `0`: enabled; all new boards use chess960.
  - `N > 0`: enabled; new boards use Chess960 when `board_id % N == 0`.
Spectator Settings
- `max-spectators` (int, 1024; -1 for no cap): Capacity limit for total active spectators at the server (summary + focus).
   Bypass is policy-controlled via `spectate.capacity_bypass` / `focus`. Bypass-focused sessions are counted outside the `max-spectators` pool.
- `spectator-visibility` (int, 0): Minimum trust tier required to spectate (summary or focus). Requests below this threshold are rejected.
- `spectator-summary-hz` (int, 2): Target max update rate (per second) for summary mode. Best-effort; updates are sent over UDP and may be batched by the main loop cadence.
- `spectator-focus-hz` (int, 20): Target max update rate (per second) for focused board updates. Best-effort over UDP.
- `spectator-max-focus-per-session` (int, 1): Focus enable switch. Supported values are 0 (disabled) and 1 (single focus). Values >1 are treated as 1.
- `spectator-summary-mode` (string, "changes"): Summary filtering mode:
  - `full`: send all ACTIVE+RESERVED boards.
  - `changes`: send only boards whose state changed since the last summary tick.

Logging Verbosity
- `log-level` (int, 0..4; default 3=INFO): Global verbosity
  (0=FATAL, 1=ERROR, 2=WARN, 3=INFO, 4=DEBUG).
  Logging uses a single global level.

Builtins for Computed and Conditional Values
- Arithmetic: `(+ a b ...)`, `(- a [b ...])`, `(* a b ...)`, `(/ a b ...)`.
- Comparison: `(= a b)` for ints, floats, or strings.
- Conditionals: `(if cond then-expr else-expr?)`.
- Environment access: `(getenv "VAR")` returns a string (empty if unset).
- Sequencing: `(do expr1 expr2 ... exprN)` evaluates in order, returns last.
- Quoting: `(quote expr)` returns `expr` without evaluating it.

Functions and Macros
- Define a function: `(defn name (arg1 arg2 ...) body...)`
  - Lexically scoped; arguments are evaluated before binding.
  - Returns the value of the last body form.
- Define a macro: `(defmacro name (arg1 arg2 ...) body...)`
  - Arguments are NOT evaluated; they are raw syntax (AST) values.
  - Body should produce an expanded form which is then evaluated at call-site.
- Example helper function:
  `(defn port-for (base offset) (+ base offset))`
  `(def port (port-for 8800 12)) ; => 8812`
- Example macro for common profile shape:
  `(defmacro simple-profile (pname pport)
     (do (defprofile pname ((def port pport)
                             (def advertise 1)
                             (def visibility 0)))))`
  `(simple-profile canary 8891)`

Examples: Functions and Macros
--------------------------------
- Helper function for ports and timeouts:
  
  `(defn port-for (base offset) (+ base offset))`
  `(defn ms->sec (ms) (/ ms 1000))`
  `(def port (port-for 8800 42))           ; => 8842`
  `(def session-timeout (ms->sec 250000))  ; => 250`

- Macro to DRY up common profiles:
  
  `(defmacro simple-profile (name p)
     (do (defprofile name ((def port p)
                            (def advertise 1)
                            (def visibility 0)
                            (def log-level 3)))))`
  
  `(simple-profile default-a 8888)`
  `(simple-profile default-b 8889)`

- Macro with inheritance (wraps defprofile :inherits):
  
  `(defmacro inherit (new base p)
     (do (defprofile new :inherits base ((def port p)))))`
  
  `(defprofile base ((def db-name "wamble_base") (def log-level 2)))`
  `(inherit canary base 8891)`

- Example of toggling global verbosity via helpers:
  
  `(defn dbg-if (flag) (if (= flag 1) 4 3))`
  `(def debug-enabled 1)`
  `(def log-level (dbg-if debug-enabled))  ; 4=DEBUG when enabled, else 3=INFO`

- Quoting and sequencing utilities:
  
  `(do (def tmp 1) (def tmp (+ tmp 1)) tmp)  ; => 2`
  `(def quoted (quote (def some-setting 1))) ; not evaluated`
  `quoted` evaluates to the AST of `(def some-setting 1)`.

Policy Rule Builtins
- `policy-allow` appends an allow rule to the config policy rule set.
- `policy-deny` appends a deny rule to the config policy rule set.
- Rules are validated/applied against the shared global policy store during
  startup/reload (see docs/database.txt).
- Argument order:
  - `policy-allow`: `(policy-allow identity action resource permission-level reason? policy-version? not-before? not-after? context-key? context-value?)`
  - `policy-deny`: `(policy-deny identity action resource reason? policy-version? not-before? not-after? context-key? context-value?)`
- Defaults:
  - `identity="*"`, `reason=""`, `policy-version="v1"`,
    `permission-level=0`, `not-before=0`, `not-after=0`.
- Identity selectors:
  - `*` for all identities.
  - `identity:<id>` or `id:<id>` for a specific global identity id.
  - `pubkey:<64-hex>` or raw `<64-hex>` to resolve/create identity by public key.
  - `tag:<name>` to target all identities in `global_identity_tags` for that tag.
- Rule fields:
  - `action`: policy action key.
  - `resource`: policy resource key.
  - `permission-level`: numeric allow level used by allow decisions.
  - `not-before` / `not-after`: unix epoch seconds mapped to
    `not_before_at` / `not_after_at`.
  - `context-key` / `context-value`: optional context predicate values.
  - `reason` / `policy-version`: metadata persisted with the rule.
- Resource mapping:
  - Generic resources are stored with `scope='*'`.
  - For `action="trust.tier"`, resources `*`, `profile:<name>`,
    `profile_group:<group>` map to `resource='tier'` and matching `scope`.
  - Scoped form `<scope>::<resource>` maps directly to DB `scope` and
    `resource`.
- Runtime action/resource registry:
  - `trust.tier`: query resource `tier`; rules should use resource
    `*`, `profile:<name>`, or `profile_group:<group>`.
  - `profile.discover.override`: query resources `profile:<name>`,
    `profile_selector:<group>`, `*`.
  - `protocol.ctrl`: query resources
    `player_move`, `board_update`, `list_profiles`, `profile_info`, `error`,
    `server_notification`, `client_goodbye`, `spectate_game`,
    `spectate_update`, `login_request`, `logout`, `login_success`,
    `login_failed`, `login_challenge`, `get_player_stats`, `player_stats_data`,
    `get_profile_info`, `profiles_list`, `spectate_stop`, `get_legal_moves`,
    `legal_moves`, `get_leaderboard`, `leaderboard_data`, `submit_prediction`,
    `get_predictions`, `prediction_data`, `get_profile_tos`,
    `profile_tos_data`, `accept_profile_tos`, `get_active_reservations`,
    `active_reservations_data`, `unknown`.
  - `game.move`: query resources `play`, `legal`.
  - `game.mode`: query resource `view`.
  - `spectate.access`: query resource `view`; context `mode=summary|focus`.
  - `spectate.capacity_bypass`: query resource `focus`.
  - `rate_limit.bypass`: query resource `request`; context
    `ctrl=<control-resource>`.
  - `stats.read`: query resource `player`; target policy context should be
    identity-tag oriented (for example `target.identity_tag=<tag>`), consistent
    with tag-based policy/treatment segmentation.
  - `leaderboard.read`: query resource `global`.
  - `prediction.write`: query resources `self`, `streak`, `gated`.
    Context includes `board=<board-id>` when available. The resolved
    `permission-level` is treated as the maximum writable prediction depth.
  - `prediction.read`: query resource `tree`; context `depth=<n>`.
    The rule permission level is treated as the maximum readable depth cap.
- Example:
  `(policy-allow "identity:42" "trust.tier" "profile:private" 5 "manual grant")`
  `(policy-deny "tag:contractors" "profile.discover.override" "profile_selector:internal" "hide internal profiles")`

Custom Policy Evaluator (optional)
- You may define `(defn policy-eval ...)` for dynamic policy decisions.
- Evaluator call signature used by `config_policy_eval(...)`:
  `(policy-eval identity action resource profile-name profile-group context-key context-value now-epoch-seconds)`
- Accepted return values:
  - integer `N` (`N >= 0`) -> allow with permission level `N`
  - string `"allow"` or `"deny"`
  - list form `(allow [permission-level] [reason] [policy-version])`
  - list form `(deny [reason] [policy-version])`
- If missing or malformed, `config_policy_eval(...)` reports no decision/error;
  final allow/deny behavior then follows the caller's own default path.
- Runtime authorization paths use DB policy resolution
  (`resolve_policy_decision`) as the source of truth.

Treatment Groups
- DSL forms:
  - `(treatment-group group-key priority)`
  - `(treatment-default group-key)`
  - `(treatment-assign identity-selector profile-scope group-key priority predicate...)`
  - `(treatment-edge source-group target-group)`
  - `(treatment-tag group-key tag-name)`
  - `(treatment-feature group-key hook-name key value)`
  - `(treatment-context group-key hook-name key value)`
  - `(treatment-behavior group-key hook-name key value)`
  - `(treatment-meta group-key hook-name key value)`
  - `(treatment-payload group-key ctrl-alias key value)`
- `predicate` form:
  - `(match fact-key op value)`
- Supported ops: `eq`, `ne`, `gt`, `gte`, `lt`, `lte`, `exists`, `absent`
- Resolution model:
  config DSL materializes into DB rows, sessions persist their assigned
  `treatment_group_key`, and runtime consumers query actions by hook.
- Outputs:
  - `tag`: persistent `global_identity_tags` upsert
  - `feature`, `context`, `behavior`, `meta`: hook-scoped outputs returned by
    treatment action lookup
  - `payload`: ctrl-scoped outputs persisted and queryable server-side.
    `ctrl-alias` accepts symbolic ctrl names (for example `prediction_data`) or
    `*`. Runtime gameplay/view paths do not emit payload outputs directly onto
    the client wire protocol.
- Hook keys consumed by runtime:
  Policy action keys and treatment hook keys are separate surfaces. Prediction
  mutation policy uses `prediction.write`; the treatment hook remains
  `prediction.submit`.
  - `prediction.submit`
    - feature: `prediction.gated`
    - behavior: `prediction.max_pending`, `prediction.points.multiplier`,
      `prediction.points.bonus`, `prediction.penalty.multiplier`
  - `prediction.read`
    - behavior: `prediction.view_depth_cap`
  - `scoring.apply`
    - feature: `scoring.disable`
    - behavior: `payout.multiplier`, `payout.bonus`
  - `rating.adjust`
    - feature: `rating.disable`
    - behavior: `rating.delta`, `rating.multiplier`
  - `board.assignment`
    - feature: `board.assignment.disable`
    - behavior: `board.score.multiplier`, `board.score.bonus`
  - `trust.resolve`
    - behavior: `trust.tier.set`, `trust.tier.delta`, `trust.tier.min`
  - `<ctrl-alias>` payload scope
    - payload: arbitrary keys with string/int/double/bool values (server-side)

- Runtime fact surface:
  - Gameplay/view treatment hooks use a fixed fact contract documented in
    `docs/treatment_gameplay_view.txt`.

- Conditional by env:
  `(def db-host (if (= (getenv "WAMBLE_ENV") "production") "prod.db" "localhost"))`
- Global verbosity examples:

  `(defn dbg (flag) (if (= flag 1) 4 3))`

Profiles
- Define a profile with grouped overrides:
  `(defprofile test ((def port 8890) (def db-name "wamble_test") (def log-level 4)))`
- Metadata inside a profile:
  - `abstract` (int 0/1, default 0): Marks the profile as a config template.
    Abstract profiles are never instantiated (no listener is started) and are
    invisible to all clients regardless of `advertise`, `visibility`, or any
    `profile.discover.override` policy rule. Use for shared config base
    profiles that exist only for inheritance. Not inherited by child profiles
    (children default to `abstract=0`).
  - `advertise` (int 0/1, default 0): Whether to advertise/offer this profile
    to clients.
  - `visibility` (int, default 0): Minimum trust tier required to see this profile in
    LIST_PROFILES. The server compares this to identity-scoped trust resolved
    for the requesting token/profile.
  - `db-isolated` (int 0/1): If 1, this profile must not share the same
    database (host/user/name) with any other advertised profile that also
    sets `db-isolated=1`. Conflicts cause startup to fail with a fatal error.
  - `profile-group` (string, optional): Logical selector used by permission
    policy rules (resource form `profile_group:<name>`).
  
- Explicit inheritance:
  `(defprofile canary :inherits base ((def port 8891) (def log-level 4)))`
  Inherits all values from profile `base`, then applies listed overrides.

Macros
- Optional helpers like `simple-profile` can reduce repetition; there is no
  db-root concept.

Defaults and Precedence
- Unspecified options use the compiled-in defaults listed above.
- Top-level `def` sets process-wide defaults; `defprofile` overrides values
  for that profile only.
- Inheritance copies all base values and metadata (including `advertise`,
  `visibility`, and `db-isolated`) unless overridden in the child profile.

Running with a Profile
- Pass `--profile <name>` to apply that profile at startup.
- Multiple profiles can be defined; the server exposes them via config APIs
  for multi-profile runtime integrations.

Concurrent Profiles
- When multiple profiles are defined with `advertise=1`, the server starts
  one listener per profile on its configured `port`, enforcing unique UDP
  ports.
- If websocket is enabled for multiple advertised profiles, effective websocket
  TCP ports must also be unique (effective port = `websocket-port` when set,
  otherwise `port`).
- On `SIGHUP`, config always reloads. Listener reconciliation only happens when
  profile topology changes, such as added/removed listeners or UDP port
  changes.
- Validation is strict: if two advertised profiles conflict (duplicate UDP
  ports or duplicate effective websocket TCP ports when enabled), the server
  terminates with a fatal error.
- DB model: Profiles may use distinct DB settings; each listener thread manages
  its own connection. If two advertised profiles both set `db-isolated=1` they
  must not point to the same DB.

Config Reload and Exec Handoff
- `SIGHUP`: rereads the configuration file without restarting the process.
  - Config is always reparsed and validated.
  - Non-socket runtime config updates are applied in place when listener
    topology is unchanged.
  - Listener reconciliation only happens when profile topology changes, such as
    added/removed listeners or UDP port changes.
  - Reload safety: if config or global policy validation fails, the running
    config remains active.
  - Restore source: runtime loads the newest `result='active'` entry from
    `global_runtime_config_revisions` joined to
    `global_runtime_config_blobs`; if DB restore is unavailable, runtime uses
    the in-memory snapshot.
- `SIGUSR2` (POSIX only): triggers an exec-based hot reload. The server exports
  active listener sockets (`WAMBLE_PROFILES_INHERITED`) and optional state file
  paths (`WAMBLE_STATE_FILE`/`WAMBLE_STATE_FILES`) before re-execing itself.
- Friendlier admin controls:
  - The server always writes a PID file to `wamble.pid` under `state-dir` (or
    `/tmp` on POSIX when `state-dir` is unset).
  - `build/wamble --reload` sends `SIGHUP` to the PID found in that file.
  - `build/wamble --hot-reload` sends `SIGUSR2` to the PID found in that file.
- Hot reload sets `WAMBLE_HOT_RELOAD=1` in the environment to instruct the new
  process to adopt inherited sockets and reload persisted state via
  `state_load_from_file`.
- When no profiles are active, the main listener socket is inherited via
  `WAMBLE_INHERITED_SOCKFD`. With profiles, `profile_runtime` handles socket
  adoption per advertised profile.
- After a successful exec reload, temporary state files are removed and
  environment markers are cleared.

Profile Runtime Notes
- `profile_runtime` owns one thread per advertised profile, mirroring each
  profile's `WambleConfig` and keeping per-thread `libpq` connections.
- `start_profile_listeners` is invoked if at least one profile is defined;
  otherwise the main thread hosts the default listener using process-wide
  defaults.
- `SIGHUP` therefore acts as config reload first, with listener reconciliation
  as a conditional sub-step rather than the default meaning of reload.
- During `SIGUSR2` exec reloads, profile threads mark themselves ready so only
  sockets that acknowledged export participate in the restart.

Visibility and Discovery
- LIST_PROFILES returns only advertised profiles whose `visibility` is less
  than or equal to the requesting client's resolved trust tier (see
  docs/database.txt), unless overridden by `profile.discover.override`
  policy. `profile.discover.override` can explicitly allow or deny discovery,
  including for unadvertised profiles.
- To activate a non-advertised profile listener, add an explicit
  `profile.discover.override` allow rule for that profile or selector.
- GET_PROFILE_INFO applies the same discovery authorization and returns
  `NOTFOUND;<name>` when not discoverable.

Notes
- Strings must be quoted. Numbers parse as integers unless they contain a dot.
- Unknown names in `def` are ignored by the engine unless referenced by code; 
  they remain available for user logic inside the config.
