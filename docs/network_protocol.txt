Wamble Network Protocol
=======================

Overview
- Transport: UDP, non-blocking sockets.
- Additional transport adapter: a WebSocket bridge (TCP) tunnels Wamble UDP
  wire packets as binary websocket frames.
- Topology: Single server, multiple listeners (one per profile).
- Reliability: Application-level ACK with retries.
- Encoding: Base64url for tokens (logs/URLs).

Message Layout
- Transport: UDP with a custom header and a variable payload.
- WebSocket transport: each binary websocket frame carries one full Wamble
  wire packet (header + payload) with identical byte layout.
- The `WambleMsg` struct in the source code is a container for message
  data, not a direct representation of the on-the-wire format.
- All packets share a common header, followed by a payload specific to the
  message's control code (`ctrl`).
- Common header (all packets):
  - `ctrl` (uint8): Control code for packet type.
  - `flags` (uint8): packet/mode flags (bitmask).
  - `version` (uint8): Protocol version (currently 1).
  - `reserved` (uint8): Must be 0.
  - `token` (uint8[16]): Session token.
  - `board_id` (uint64): Board context (0 if not applicable).
  - `seq_num` (uint32): Reliable sequencing number.
  - `payload_len` (uint16): Length of the payload data that follows the header.
- Endianness:
  - `board_id`: Big-endian (8 bytes).
  - `seq_num`, `payload_len`: Network byte order (big-endian).
- Token URL form: 22-char base64url string (no padding).
- `flags` bits:
  - `0x80` (`WAMBLE_FLAG_UNRELIABLE`): packet is unreliable.
  - `0x01` (`WAMBLE_FLAG_BOARD_IS_960`): board in this packet is Chess960.
  - `0x02` (`WAMBLE_FLAG_MODE_FILTER_CHESS960`): spectator summary filter includes Chess960 boards.
  - `0x04` (`WAMBLE_FLAG_MODE_FILTER_STANDARD`): spectator summary filter includes standard boards.
  - `0x08` (`WAMBLE_FLAG_EXT_PAYLOAD`): this packet payload carries an
    extension trailer (non-fragment envelope packets).
  - `0x10` (`WAMBLE_FLAG_FRAGMENT_PAYLOAD`): payload is encoded using the
    Fragment Payload Envelope.

Extension Trailer (reserved)
- Reserved typed key/value trailer for protocol metadata.
- It is present only when a sender explicitly sets `WambleMsg.ext_count > 0`
  and provides `ext[]` fields.
- When `WAMBLE_FLAG_EXT_PAYLOAD` is set, the message payload is:
  - `<base payload bytes>`
  - `<extension body>`
  - footer: `0x57 0x58 <uint16 extension_body_len_be>`
- Extension body format (version 1):
  - `uint8 version` (`1`)
  - `uint8 field_count`
  - Repeated fields:
    - `uint8 key_len`, `char key[key_len]`
    - `uint8 value_type` (`string=1`, `int=2`, `double=3`, `bool=4`)
    - value:
      - string: `uint16 str_len`, `char str[str_len]`
      - int: `int64` big-endian
      - double: IEEE754 `uint64` bit pattern, big-endian
      - bool: `uint8` (`0` or `1`)
- Current capability metadata keys:
  - `session.caps` (`int` bitmask): server-advertised UI/session actions.
  - `profile.caps` (`int` bitmask): per-profile discovery actions.
  - `prediction.source` (`string`): `tree` or `moves` for prediction reads.
- Current bitmasks:
  - `session.caps`
    - `0x00000001`: attach identity
    - `0x00000002`: logout
    - `0x00000004`: move interaction
    - `0x00000008`: stats
    - `0x00000010`: leaderboard
    - `0x00000020`: spectate summary
    - `0x00000040`: spectate focus
    - `0x00000080`: prediction submit
    - `0x00000100`: prediction read
  - `profile.caps`
    - `0x00000001`: join/profile attach
    - `0x00000002`: terms available

Fragment Payload Envelope (version 1)
- The sender uses this envelope when an eligible control payload does not fit
  in a single packet (`WAMBLE_MAX_PAYLOAD` bytes).
- Sender default for eligible controls: fragment instead of truncate.
- Receivers only decode the fragment envelope when
  `WAMBLE_FLAG_FRAGMENT_PAYLOAD` is set.
- Eligible controls (exhaustive):
  - `SERVER_HELLO`
  - `BOARD_UPDATE`
  - `PROFILE_INFO`
  - `ERROR`
  - `SERVER_NOTIFICATION`
  - `SPECTATE_UPDATE`
  - `LOGIN_CHALLENGE`
  - `LOGIN_FAILED`
  - `PLAYER_STATS_DATA`
  - `PROFILES_LIST`
  - `LEGAL_MOVES`
  - `LEADERBOARD_DATA`
  - `PREDICTION_DATA`
  - `PROFILE_TOS_DATA`
- Fragment wire payload format:
  - `uint8 fragment_version` (`1`)
  - `uint8 hash_algo` (`1` = BLAKE2b-256)
  - `uint16 chunk_index_be`
  - `uint16 chunk_count_be`
  - `uint32 total_len_be`
  - `uint32 transfer_id_be`
  - `uint8 fragment_hash[32]` (BLAKE2b-256 over the full reassembled payload)
  - `uint16 chunk_len_be`
  - `uint8 chunk_bytes[chunk_len]`
- All chunks in one transfer share `hash_algo`, `total_len_be`,
  `transfer_id_be`, and `fragment_hash`.
- Chunk payload max is 1154 bytes (`WAMBLE_MAX_PAYLOAD - 46`).
- The chunk byte stream is the exact original payload byte stream (including
  any extension trailer bytes if present).

Control Codes
- `0x01` CLIENT_HELLO (client to server): Initial connection request.
  - Payload: empty.
- `0x02` SERVER_HELLO (server to client): Response to HELLO.
  - Payload: `char fen[]` (current board state).
  - May include extension fields: `session.caps`, `prediction.source`.
- `0x03` PLAYER_MOVE (client to server): Submit a move.
  - Payload: `uint8 uci_len; char uci[uci_len]` (max 6 bytes).
- `0x04` BOARD_UPDATE (server to client): Board state has changed.
  - Payload: `char fen[]`.
  - `flags`: server sets `0x01` when the assigned board is Chess960.
  - May include extension fields: `session.caps`, `prediction.source`.
- `0x05` ACK (bidirectional): Acknowledges a reliable packet.
  - Payload: empty. Header `seq_num` matches the packet being ACKed.
- `0x06` LIST_PROFILES (client to server): Request available profiles.
  - Payload: empty.
- `0x07` PROFILE_INFO (server to client): Response to GET_PROFILE_INFO.
  - Payload: `name;port;advertise;visibility` or `NOTFOUND;<name>`.
  - May include extension field `profile.caps`.
- `0x08` ERROR (server to client): Reports an error.
  - Payload: `uint16 code; uint8 reason_len; char reason[reason_len]`.
  - Reliable by default.
- `0x09` SERVER_NOTIFICATION (server to client): Broadcast message.
  - Payload: `char text[]`. Often sent as unreliable.
- `0x0A` CLIENT_GOODBYE (client to server): Graceful disconnect.
  - Payload: empty.
- `0x0B` SPECTATE_GAME (client to server): Request to watch a game.
  - Payload: empty.
  - `flags` (summary mode only, `board_id=0`):
    - `0x02`: include Chess960 boards.
    - `0x04`: include standard boards.
    - Omitting both bits requests an unfiltered summary stream.
  - Header semantics:
    - `board_id=0` enters summary mode. The server will push per-board
      `SPECTATE_UPDATE` packets for eligible boards (ACTIVE+RESERVED), sorted
      by recency and attractiveness. Immediate unreliable updates are sent on
      entry, followed by periodic updates at `spectate-summary-hz`.
    - `board_id>0` focuses a specific board. The server will push
      `SPECTATE_UPDATE` packets for that board only. Immediate unreliable
      update is sent on entry, followed by periodic updates at
      `spectate-focus-hz`.
- `0x16` SPECTATE_STOP (client to server): Stop spectating.
  - Payload: empty. Server acknowledges. Puts session in `IDLE` state.
- `0x0C` SPECTATE_UPDATE (server to client): Board update for spectators.
  - Header: `board_id` identifies the board for this update (authoritative).
  - Payload: `char fen[]`.
  - Typically sent as unreliable (`flags` high bit set). One packet per board.
  - Payload carries FEN data; `board_id` is always present in the header and
    clients use it to associate updates with boards.
  - `flags`: server sets `0x01` when the update is for a Chess960 board.
- `0x0D` LOGIN_REQUEST (client to server): Attach persistent identity to an
  existing anonymous session.
  - Two-step payload variants:
    - Challenge request: `uint8 public_key[32]`.
    - Signed proof: `uint8 public_key[32]; uint8 signature[64]`.
  - Signature message format:
    - ASCII context bytes: `"wamble.login.v1"`
    - `uint8 protocol_version` (`1`)
    - `uint8 token[16]`
    - `uint8 public_key[32]`
    - `uint8 challenge[32]` (from `LOGIN_CHALLENGE`)
  - Signature algorithm: Ed25519.
  - Server behaviour:
    - Unsigned variant issues `LOGIN_CHALLENGE`.
    - Signed variant verifies challenge freshness and signature, then links the
      session token to the persistent identity.
  - Requires a valid session token from a prior `CLIENT_HELLO`.
- `0x0E` LOGIN_CHALLENGE (server to client): One-time challenge for login proof.
  - Payload: `uint8 challenge[32]`.
  - Sent reliably.
- `0x0F` LOGOUT (client to server): Disassociate persistent identity.
  - Payload: empty.
- `0x10` LOGIN_SUCCESS (server to client): Successful login.
  - Payload: empty.
  - May include extension fields: `session.caps`, `prediction.source`.
- `0x11` LOGIN_FAILED (server to client): Failed login.
  - Payload: `uint16 code; uint8 reason_len; char reason[reason_len]`.
- `0x12` GET_PLAYER_STATS (client to server): Request player statistics.
  - Payload: empty.
- `0x13` PLAYER_STATS_DATA (server to client): Player statistics.
  - Payload: `double score_be; uint32 games_played_be; uint32 chess960_games_played_be`.
  - Note: Rating is tracked separately from score; this payload carries score and game counters.
- `0x14` GET_PROFILE_INFO (client to server): Request info for a specific profile.
  - Payload: `uint8 name_len; char name[name_len]`.
- `0x15` PROFILES_LIST (server to client): Response to LIST_PROFILES.
  - Payload: `char csv[]` (comma-separated profile names).
- `0x17` GET_LEGAL_MOVES (client to server): Request legal moves for a square.
  - Payload: `uint8 square_index` (0-63, 0 = a1).
  - Header: `board_id` must reference the board assignment for the player.
  - Authorization: server validates the requesting token owns the reservation;
    otherwise the response is an empty `LEGAL_MOVES` packet.
- `0x18` LEGAL_MOVES (server to client): Response to GET_LEGAL_MOVES.
  - Payload: `uint8 square_index; uint8 move_count; struct { uint8 from; uint8 to; int8 promotion; } moves[move_count]`.
  - `move_count` is capped at 218 and only includes moves originating from `square_index`.
  - Promotion is encoded as ASCII lowercase (`'q'`, `'r'`, `'b'`, `'n'`) or 0 for non-promotions.
  - Typically sent reliably so clients can pre-populate hover highlights.
- `0x19` GET_LEADERBOARD (client to server): Request leaderboard snapshot.
  - Payload: `uint8 leaderboard_type; uint8 limit`.
  - `leaderboard_type`: `1=score`, `2=rating`.
  - `limit`: defaults to 10 if 0.
- `0x1A` LEADERBOARD_DATA (server to client): Leaderboard response.
  - Payload:
    - `uint8 leaderboard_type`
    - `uint8 count`
    - `uint32 self_rank_be` (0 when requester has no known rank)
    - `count * struct { uint32 rank_be; uint64 session_id_be; uint64 score_bits_be; uint64 rating_bits_be; uint32 games_played_be; }`
  - `score_bits_be` and `rating_bits_be` are IEEE-754 double bit patterns in big-endian order.
  - Maximum entries per packet: 16.
- `0x1B` SUBMIT_PREDICTION (client to server): Submit a prediction for the next
  half-move on `board_id`, or for the next half-move after
  `parent_prediction_id_be` when replying on a prediction branch.
  - Payload: `uint8 uci_len; char uci[uci_len]; uint64 parent_prediction_id_be`.
  - `parent_prediction_id_be = 0` means a root prediction.
- `0x1C` GET_PREDICTIONS (client to server): Request prediction tree rows for a
  board.
  - Payload: `uint8 depth; uint8 limit`.
  - `depth = 0` lets the server use its configured default depth cap.
  - `limit` is capped by the implementation maximum entries per packet.
- `0x1D` PREDICTION_DATA (server to client): Prediction tree response or submit
  confirmation.
  - Payload:
    - `uint8 count`
    - `count * struct {`
      `uint64 id_be;`
      `uint64 parent_id_be;`
      `uint8 token[16];`
      `uint64 points_bits_be;`
      `uint16 target_ply_be;`
      `uint8 depth;`
      `uint8 status;`
      `uint8 uci_len;`
      `char uci[6];`
      `}`
  - `status`: `0=pending`, `1=correct`, `2=incorrect`, `3=expired`.
  - `points_bits_be` is an IEEE-754 double bit pattern in big-endian order.
  - Maximum entries per packet: 16.
- `0x1E` GET_PROFILE_TOS (client to server): Request ToS text for a profile.
  - Payload: `uint8 name_len; char name[name_len]`.
  - Same authorization rules as GET_PROFILE_INFO: profile must be discoverable
    by the requesting session's trust tier.
- `0x1F` PROFILE_TOS_DATA (server to client): Response to GET_PROFILE_TOS.
  - Payload is the ToS text bytes.
  - If payload size exceeds `WAMBLE_MAX_PAYLOAD`, sender uses the
    Fragment Payload Envelope.
  - If the profile is not found or not accessible: `NOTFOUND;<name>`.
  - Sent reliably.

Reliability
- Reliable messages are sent with a sequence number (`seq_num`).
- Receiver must respond with an `ACK` packet with the same `seq_num`.
- Sender retries if `ACK` is not received within `timeout-ms`.
  - Retries up to `max-retries` times, with exponential backoff.
- Duplicate detection: A sliding window (`W=1024`) of received sequence numbers is
  maintained per session to drop duplicate or old packets.

Unreliable Messages
- Marked by setting the high bit of the `flags` field (`0x80`).
- Sent "fire-and-forget" without expecting an `ACK`.
- No sequencing or duplicate filtering is applied by the receiver.
- `seq_num` should be set to 0 by the sender.
- Used for non-critical, high-frequency updates (e.g., `SPECTATE_UPDATE`).

Spectator Policy
- Visibility: spectating requires resolved trust >= `spectate-visibility`
  from global policy rules.
- Capacity: the server may cap the total number of active spectators via
  `max-spectators`. Bypass-focused sessions are counted outside
  the `max-spectators` pool.
- Summary modes:
  - `full`: send all ACTIVE+RESERVED boards.
  - `changes`: send only boards changed since the last summary tick.
- Ordering: after filtering, boards are ordered by most recent play and then
  by an internal "attractiveness" score aligned with board assignment logic.

Sessions
- A session is identified by a `token` and the client's address:port.
- Session table size is `max-client-sessions` per listener.
- Request throughput can be limited per token using
  `rate-limit-requests-per-sec`; bypass is policy-controlled via
  `rate_limit.bypass` with resource `request`.
- NAT rebinding: The address:port for a session can be updated if a valid
  packet arrives with the same token and a valid `seq_num`.
- Idle sessions are removed after `session-timeout` seconds.

Profiles
- The server runs one UDP listener thread per advertised profile.
- `LIST_PROFILES` returns:
  - advertised profiles where `profile.visibility <= resolved_trust`;
  - profiles explicitly allowed by `profile.discover.override` policy.
- `GET_PROFILE_INFO` requests details for a named profile and uses the same discovery authorization as `LIST_PROFILES`.

WebSocket Bridge
- Endpoint: `ws://<host>:<websocket-port><websocket-path>`
  - Controlled by config keys: `websocket-enabled`, `websocket-port`,
    `websocket-path`.
  - If `websocket-port` is `0`, the server uses `port` for websocket TCP.
- Handshake:
  - HTTP `GET` to `websocket-path`.
  - Requires standard websocket upgrade headers (`Upgrade`, `Connection`,
    `Sec-WebSocket-Key`, `Sec-WebSocket-Version: 13`).
- Framing:
  - Client -> server: masked binary frames (`opcode=0x2`) containing one
    Wamble packet each.
  - Server -> client: unmasked binary frames containing one Wamble packet each.
  - Data transport is binary-only; text frames are not supported.
  - Continuation data frames are supported for fragmented binary messages.
  - Ping/Pong and Close control frames are supported.
  - Fragmentation is supported for binary messages (`opcode=0x2` then
    `opcode=0x0` continuations until `FIN=1`).
  - Control frames must remain unfragmented.
  - Frame payload limit is currently 4096 bytes.
- Semantics:
  - Control codes and packet bytes are unchanged.
  - Reliability/ACK and sequence semantics are enforced by the application
    protocol, independent of websocket/TCP transport reliability.
  - Gateway optimization: for reliable server->client packets, the bridge
    emits protocol ACKs locally after successful websocket frame write.
  - The websocket layer is strictly a transport adapter over the existing
    protocol.
  - The adapter creates a loopback UDP socket per websocket client; server-side
    session address/port therefore reflects the local bridge socket (not the
    browser's remote IP/port).
