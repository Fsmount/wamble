Wamble Prediction
=================

Overview
- Lets sessions submit move predictions against the next half-move on a board,
  or a later half-move when the prediction references a parent branch.
- Prediction scoring is separate from normal board pot scoring.
- Predictions may reference an earlier prediction, which lets the server expose
  a prediction tree for the active board.

Modes
- `prediction-mode = 0`: disabled.
- `prediction-mode = 1`: only the currently reserved mover may predict their
  next move.
- `prediction-mode = 2`: allowed sessions may predict the next half-move and
  build a streak across correct predictions.
- `prediction-mode = 3`: same as mode 2, plus deterministic session gating via
  `prediction-gated-percent`.

Scoring
- Correct predictions award `prediction-base-points` in mode 1.
- Correct predictions award
  `prediction-base-points * prediction-streak-multiplier^streak_before`
  in modes 2 and 3, capped by `prediction-streak-cap`.
- Incorrect predictions apply `prediction-penalty-incorrect` as a negative
  prediction score delta when configured.
- Prediction rewards update `player->prediction_score`, not `player->score`.
- Session aggregates are stored in `sessions.total_prediction_score`.

Policy
- Write access uses policy action `prediction.write`.
- Write resources are `self`, `streak`, or `gated`, depending on mode.
- Write requests pass `board=<board-id>` as context when the runtime has it.
- For `prediction.write`, the resolved policy `permission_level` is treated as
  the maximum writable tree depth.
- Tree reads use policy action `prediction.read` with resource `tree`.
- Tree reads pass `depth=<requested-depth>` as context.
- For `prediction.read`, the resolved policy `permission_level` is treated as
  the maximum readable tree depth.
- Root submissions require writable depth `0`.
- Reply submissions require writable depth `parent_depth + 1`.
- Write and read are independent; a session may be allowed to submit without
  being allowed to read the tree.

Duplicate Policy
- Per-player-per-parent duplicate: a session may have at most
  `prediction-max-per-parent` pending predictions per parent branch on a board
  (default `1`). Predictions are immutable — once submitted, the move cannot be
  changed. Attempting to exceed this cap returns `PREDICTION_ERR_DUPLICATE`
  with one existing prediction ID in `out_prediction_id`.
- Move-identical duplicate: if another session has already submitted a pending
  prediction with the same UCI move at the same parent on the same board, and
  `prediction-enforce-move-duplicate != 0`:
  - If the submitting session has prediction read access: rejected with
    `PREDICTION_ERR_DUPLICATE_MOVE` (the session could have seen it).
    The server responds with `PREDICTION_DATA` containing the existing
    prediction row.
  - If the submitting session lacks read access: accepted. Independent
    conclusions from sessions that cannot observe each other are valid.
    The server retries with `WAMBLE_PREDICTION_SKIP_MOVE_DUP` internally.

Tree Semantics
- Each prediction may reference `parent_prediction_id`.
- Root predictions have no parent.
- Root predictions target the board's next half-move.
- Child predictions target the next half-move after their parent target.
- Descendants of an incorrect or expired branch are expired automatically.

Tree Reads
- Tree reads return flat rows with `id`, `parent_id`, `board_id`,
  predictor token, move, status, target ply, depth, awarded points, and
  creation time.
- Consumers reconstruct the tree by linking `parent_id` to `id`.
- The effective depth limit is the minimum of:
  - `prediction-view-depth-limit`
  - requested depth
  - policy `permission_level` for `prediction.read`

Pending Limit Scope
- `prediction-max-pending` applies at board/readability-depth scope so sessions
  sharing visible prediction context are constrained by the same pending budget.
- Branch-level duplicate controls still apply independently via
  `prediction-max-per-parent`.

Compatibility
- Treatment hooks for prediction scoring and gating still use the existing
  `prediction.submit` hook name.

Persistence
- Prediction submissions are stored in `predictions`.
- Prediction resolution updates `status`, `points_awarded`, and `resolved_at`.
- Resolved prediction totals are rolled into `sessions.total_prediction_score`.

API
- `prediction_manager_init()`.
- `prediction_submit()`, `prediction_submit_with_parent()`.
  - `prediction_submit_with_parent()` outputs the existing prediction ID in
    `out_prediction_id` on both `PREDICTION_ERR_DUPLICATE` and
    `PREDICTION_ERR_DUPLICATE_MOVE`.
  - The `trust_tier` parameter accepts `WAMBLE_PREDICTION_SKIP_MOVE_DUP`
    (`0x100`) to bypass the move-identical duplicate check.
- `prediction_resolve_move()`.
- `prediction_collect_tree()`.
- `prediction_get_view_by_id()` - authoritative read for `SUBMIT_PREDICTION`
  responses; server returns the persisted submit result by id rather than
  walking a depth-limited tree re-read.
- `prediction_max_pending_for_player()` - resolves the treatment-adjusted
  pending cap advertised via the `prediction.max_pending` extension field.
- `prediction_expire_board()`.

References
- Config (see docs/configuration.txt): `prediction-mode`,
  `prediction-base-points`, `prediction-streak-multiplier`,
  `prediction-streak-cap`, `prediction-gated-percent`,
  `prediction-penalty-incorrect`, `prediction-match-policy`,
  `prediction-max-pending`, `prediction-max-per-parent`,
  `prediction-enforce-move-duplicate`, `prediction-view-depth-limit`.
- Protocol (see docs/network_protocol.txt): `SUBMIT_PREDICTION`,
  `GET_PREDICTIONS`, `PREDICTION_DATA`.
  - Every `PREDICTION_DATA` packet carries `prediction.request_kind`
    (`submit`|`tree`). Submit responses additionally carry
    `prediction.submit_status` (1=created, 2=duplicate, 3=duplicate_move,
    4=rejected_invalid, 5=rejected_limit, 6=rejected_not_allowed,
    7=rejected_not_found, 8=rejected_disabled).
  - Prediction failures respond with `PREDICTION_DATA` (count=0), not generic
    `ERROR`. On duplicate with read access, the existing prediction row is
    returned (count=1) with `prediction.submit_status` of 2 or 3.
  - `SERVER_HELLO`, `BOARD_UPDATE`, and `LOGIN_SUCCESS` carry
    `prediction.max_pending` when submit capability is set; clients must use
    this value rather than any compile-time default.
- Scoring (see docs/scoring.txt): normal board pot scoring remains separate.
- Database (see docs/database.txt): tables `predictions`, `sessions`.
